# Table Update APIs

This document describes the new table update APIs that allow you to update existing tables with enhanced column definitions, table descriptions, and metadata while preserving existing data when possible.

## Overview

The table update APIs provide two main endpoints:

1. **PUT** `/workflow/table/{table_id}` - Full table update (replace all data)
2. **PATCH** `/workflow/table/{table_id}` - Partial table update (selective updates)

Both endpoints support enhanced column definitions and table descriptions generated by LLM, and can either update existing tables or create new ones if they don't exist.

## API Endpoints

### PUT /workflow/table/{table_id}

**Purpose**: Full table update or creation

**Behavior**:
- If table exists: Completely replaces all table data and metadata
- If table doesn't exist: Creates a new table with the specified ID
- Updates all aspects: columns, descriptions, enhanced metadata
- Increments entity version for change tracking

**Use Cases**:
- Complete table schema redesign
- Migrating tables between environments
- Full table replacement scenarios

### PATCH /workflow/table/{table_id}

**Purpose**: Partial table update

**Behavior**:
- Only updates the specified aspects of the table
- Preserves existing data and metadata unless explicitly updated
- Supports selective updates using boolean flags
- Increments entity version for change tracking

**Use Cases**:
- Adding new columns while preserving existing ones
- Updating table descriptions without changing schema
- Enhancing metadata without structural changes
- Incremental schema evolution

## Request Models

### UpdateTableRequest

```python
class UpdateTableRequest(BaseModel):
    dataset_id: str
    schema: SchemaInput
    update_description: Optional[bool] = True
    update_columns: Optional[bool] = True
    update_enhanced_metadata: Optional[bool] = True
    preserve_existing_metadata: Optional[bool] = False
```

**Fields**:
- `dataset_id`: ID of the dataset containing the table
- `schema`: Table schema definition (same as AddTableRequest)
- `update_description`: Whether to update table description
- `update_columns`: Whether to update column definitions
- `update_enhanced_metadata`: Whether to update enhanced metadata
- `preserve_existing_metadata`: Whether to preserve existing metadata not in the update

## Enhanced Features

### Enhanced Column Definitions

The update APIs support enhanced column definitions that include:

- **Business-friendly names and descriptions**
- **Usage type classification** (dimension, measure, attribute, identifier, timestamp, flag, metadata, calculated)
- **Data quality checks and business rules**
- **Privacy classification** (public, internal, confidential, restricted)
- **Aggregation and filtering suggestions**
- **Related business concepts**
- **Example values and patterns**

### LLM-Generated Insights

All updates leverage LLM processing to generate:

- Semantic table descriptions
- Business purpose and use cases
- Key relationships and data lineage
- Performance considerations
- Access patterns and data retention policies

## Business Rules and Constraints

### Domain Status Requirements

- Tables can only be updated in domains with status `draft` or `draft_ready`
- Active domains require promotion to draft status before updates

### Table Limits

- Maximum 4 tables per domain
- Updates that would exceed this limit are rejected

### Naming Constraints

- Table names must be unique within a domain
- Updates that would create duplicate names are rejected

### Versioning

- All updates increment the `entity_version` field
- Changes are tracked with `modified_by` and `updated_at` timestamps

## Error Handling

### Common Error Scenarios

1. **Table Not Found** (404)
   - PUT: Creates new table
   - PATCH: Returns 404 error

2. **Domain Status Invalid** (400)
   - Domain must be in draft or draft_ready status

3. **Duplicate Table Name** (400)
   - New name conflicts with existing table in domain

4. **Table Limit Exceeded** (400)
   - Domain already has maximum 4 tables

5. **Dataset Not Found** (404)
   - Specified dataset_id doesn't exist

### Error Response Format

```json
{
    "detail": "Error description",
    "status_code": 400
}
```

## Best Practices

### When to Use PUT vs PATCH

- **Use PUT** for:
  - Complete table replacements
  - Schema migrations
  - Creating tables with specific IDs

- **Use PATCH** for:
  - Incremental updates
  - Metadata enhancements
  - Selective field updates

### Metadata Preservation

- Set `preserve_existing_metadata: true` to keep custom metadata
- Use selective update flags to minimize data loss
- Test updates in development environments first

### Performance Considerations

- Column updates recreate all columns (can't update individual columns)
- Large tables may take longer to process
- Consider batching multiple updates

## Migration Guide

### From Old APIs

If you were using custom update logic:

1. Replace custom update code with the new endpoints
2. Use PUT for full replacements, PATCH for partial updates
3. Leverage the enhanced metadata features
4. Update your request models to use `UpdateTableRequest`

### Testing Updates

1. Test updates in draft domains first
2. Verify metadata preservation with `preserve_existing_metadata: true`
3. Check entity version increments
4. Validate enhanced column definitions

## Troubleshooting

### Common Issues

1. **Update Fails with 400 Error**
   - Check domain status (must be draft or draft_ready)
   - Verify table name uniqueness
   - Check table limits

2. **Metadata Lost After Update**
   - Ensure `preserve_existing_metadata: true` is set
   - Check which update flags are enabled

3. **Columns Not Updated**
   - Verify `update_columns: true` is set
   - Check schema input format

4. **Enhanced Metadata Missing**
   - Ensure `update_enhanced_metadata: true` is set
   - Verify domain context is provided

### Debug Information

- Check logs for detailed error messages
- Verify request payload format
- Confirm database constraints and relationships
- Test with minimal update requests first
