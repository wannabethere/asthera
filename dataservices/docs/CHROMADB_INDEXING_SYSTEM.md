# ChromaDB Indexing System

## Overview

The ChromaDB Indexing System is a comprehensive solution for automatically indexing project metadata, table descriptions, and database schemas into ChromaDB for semantic search and retrieval. It integrates with the async job queue system to process MDL (Model Definition Language) files generated by LLM definition generation.

## Architecture

### Components

1. **ChromaDB Indexing Job Handler** (`job_handlers.py`)
   - Processes MDL files from LLM definition generation
   - Uses existing indexing components: ProjectMeta, TableDescription, DBSchema
   - Updates project metadata with indexing results

2. **Indexing Components**
   - **ProjectMeta**: Indexes project-level metadata
   - **TableDescription**: Indexes table descriptions and business context
   - **DBSchema**: Indexes database schema information

3. **Integration Points**
   - **Post-Commit Workflow**: Automatically triggers indexing after project commits
   - **Manual Triggering**: API endpoint for manual indexing
   - **Job Queue**: Asynchronous processing with status tracking

## Job Types

### CHROMADB_INDEXING
- **Purpose**: Index project data into ChromaDB using MDL files
- **Priority**: Medium (1)
- **Dependencies**: Requires LLM definitions to be generated first
- **Input**: Project ID with MDL file path in metadata
- **Output**: Indexing results and updated project metadata

## API Endpoints

### Manual Indexing Trigger
```http
POST /api/job-queue/chromadb-indexing/{project_id}
```

**Parameters:**
- `project_id` (path): ID of the project to index
- `user_id` (query, optional): User ID for tracking
- `session_id` (query, optional): Session ID for tracking

**Response:**
```json
{
  "message": "ChromaDB indexing job submitted successfully",
  "job_id": "job_123",
  "project_id": "project_456"
}
```


**Parameters:**
- `project_id` (query): ID of the committed project
- `user_id` (query, optional): User ID for tracking

## Workflow

### 1. Project Commit
When a project is committed:
1. Post-commit workflow generates LLM definitions
2. Creates MDL file with enhanced table definitions
3. Automatically submits ChromaDB indexing job

### 2. ChromaDB Indexing Job
The indexing job:
1. Retrieves project details from database
2. Reads MDL file path from project metadata
3. Loads and parses MDL data
4. Initializes indexing components
5. Processes project metadata, table descriptions, and database schema
6. Updates project metadata with indexing results

### 3. Indexing Components
Each component processes different aspects:

#### ProjectMeta
- Indexes project-level metadata
- Stores project context and business domain
- Creates searchable project information

#### TableDescription
- Processes table definitions from MDL
- Extracts business purpose, use cases, relationships
- Indexes column-level business context
- Creates rich descriptions for semantic search

#### DBSchema
- Indexes technical schema information
- Processes table structures and relationships
- Creates searchable schema documentation

## Usage Examples

### Manual Indexing
```python
from app.services.job_queue_service import JobQueueService, JobType

# Submit indexing job
job_id = await job_queue_service.submit_job(
    job_type=JobType.CHROMADB_INDEXING,
    project_id="project_123",
    entity_type="project",
    entity_id="project_123",
    user_id="user_456",
    priority=1
)

# Check job status
status = await job_queue_service.get_job_status(job_id)
print(f"Job status: {status.status.value}")
```

### API Usage
```bash
# Trigger manual indexing
curl -X POST "http://localhost:8000/api/job-queue/chromadb-indexing/project_123?user_id=user_456"

# Check job status
curl "http://localhost:8000/api/job-queue/jobs/job_123/status"
```

## Error Handling

### Common Issues
1. **MDL File Not Found**: Ensure LLM definitions are generated first
2. **Project Not Found**: Verify project exists and is committed
3. **ChromaDB Connection**: Check ChromaDB configuration and connectivity
4. **OpenAI API**: Verify API key and rate limits

### Error Recovery
- Jobs automatically retry on failure (configurable retry count)
- Failed jobs can be manually retried via API
- Error details are stored in job metadata
- Project metadata is updated with error information

## Troubleshooting

### Common Problems

#### Job Stuck in PENDING
- Check if worker is running
- Verify Redis connectivity
- Check job queue configuration

#### Indexing Fails
- Verify MDL file exists and is valid JSON
- Check OpenAI API key and rate limits
- Ensure ChromaDB is accessible
- Review error logs for specific issues

#### Poor Search Results
- Verify embeddings are generated correctly
- Check document chunking strategy
- Review metadata structure
- Consider re-indexing with updated settings
