# Artifact Generation System — System Design
**Three-Artifact Pipeline: Dashboard Layout + CubeJS Assembly + n8n Workflow**

---

## Design Philosophy

Every approved dashboard group produces exactly **three artifacts** that travel together as a versioned unit. They are generated by a coordinated multi-agent system, stored in a versioned file tree (git-backed), and can be re-run independently or as a full group when upstream inputs change.

The human approved the dashboard layout at VERIFY. The two additional artifacts — CubeJS schema and n8n workflow — are **fully automated outputs** derived from the same approved spec. No second human gate. The only human interaction in this system beyond VERIFY is reviewing a diff when an existing group is being updated.

```
Approved RBSRV Spec (from VERIFY)
           ↓
  [ARTIFACT ORCHESTRATOR]  ← LangGraph, 3-agent fan-out
    ├── [LAYOUT AGENT]      → dashboard_layout_spec.json  (pass-through from VERIFY)
    ├── [CUBEJS AGENT]      → cubejs/*.js + schema.yml
    └── [N8N AGENT]         → n8n_workflow.json
           ↓
  [VERSION AGENT]           → bump version, write CHANGELOG.md, write manifest.json
           ↓
  [STORAGE AGENT]           → write to file system + git commit
           ↓
  Versioned artifact group committed
```

## Semantic Versioning Rules

Each artifact group has an independent version. Bumps are determined by the `dependency_check_node` before generation runs.

| Change Type | Bump | Trigger Signal |
|---|---|---|
| Gold table schema change (column added/removed) | MAJOR | `gold_table_schema_diff` |
| New metric or KPI added to the group | MINOR | `metric_catalog_delta` |
| New control anchor added | MINOR | `control_anchor_delta` |
| Threshold or label change | PATCH | `threshold_delta` |
| Dashboard template re-scored to different template | MAJOR | `template_id_change` |
| CubeJS pre-aggregation update only | PATCH | `preagg_update` |
| n8n workflow schedule change | PATCH | `schedule_delta` |
| Insight text refresh | PATCH | `insight_delta` |

## Component 1 — ARTIFACT ORCHESTRATOR (LangGraph)

### State Schema

```python
class ArtifactOrchestratorState(TypedDict):
    # Input
    group_queue: list[dict]          # groups to process [{group_id, layout_spec, ...}]
    current_group: dict              # the group currently being processed

    # Dependency check
    staleness_results: dict          # {group_id: {stale, reason, bump_type}}
    dependency_map: dict             # gold_table → [group_ids]

    # Generation outputs (per current_group)
    layout_artifact: dict            # pass-through from layout_spec
    cubejs_artifact: dict            # {schema_yml, cube_files, pre_aggs}
    n8n_artifact: dict               # {workflow_json, node_count, conditions}

    # Version management
    current_version: str
    new_version: str
    bump_type: str                   # major | minor | patch
    changelog_entry: str             # LLM-generated

    # Validation
    validation_results: dict         # {artifact: {passed, errors[]}}

    # Storage
    storage_paths: dict              # {artifact_name: file_path}
    git_commit_hash: str

    # Control
    phase: ArtifactPhase
    errors: list[str]
    processed_groups: list[str]
    failed_groups: list[str]
```

### Graph Topology

```
START → intake
  ↓
intake → dependency_check → scheduler
  ↓
scheduler → [for each group in priority queue]
  ↓
group_intake → [fan-out parallel]
  ├── layout_node (sync — pass-through)
  ├── cubejs_node (async — LLM generative)
  └── n8n_node    (async — LLM generative)
  ↓ [fan-in]
version_node → validation_node
  ↓ [passes validation?]
  YES → storage_node → notify_node → [next group or END]
  NO  → error_node → [retry or skip group]
```

### Node Responsibilities

**`intake_node`** — Loads `group_manifest.json` and `artifact_versions.json`. Builds the group queue from all pending group_ids. Groups are pending if they have no prior version OR a staleness signal exists.

**`dependency_check_node`** — For each group in the queue, checks `dependency_graph.json` against gold table change signals. Determines `bump_type` per group. Groups with no changes are removed from the queue.

**`scheduler_node`** — Prioritizes the queue. MAJOR bumps first (schema-breaking), then MINOR, then PATCH. Within a bump type, groups with more control anchors run first (higher compliance surface area = higher priority). Writes priority-ordered queue back to state.

**`layout_node`** — Pure pass-through. Copies the approved layout spec from `current_group.layout_spec` into `layout_artifact`. Adds `generated_at` timestamp and version tag.

**`cubejs_node`** — The generative node for CubeJS. See Agent 2 below.

**`n8n_node`** — The generative node for n8n. See Agent 3 below.

**`version_node`** — Applies semver bump, generates CHANGELOG.md entry using LLM (Prompt 6), writes new `manifest.json`.

**`validation_node`** — Validates each artifact against its schema:
- Layout spec: validates against the RBSRV Unified Dashboard Spec schema
- CubeJS: validates JS syntax + schema.yml structure + measure/dimension completeness
- n8n: validates against n8n workflow export schema (required node types present)

**`storage_node`** — Writes all artifact files to the versioned directory, updates `latest` symlink, updates `artifact_versions.json`, runs `git add + git commit` with the changelog entry as commit message.

**`notify_node`** — Emits a completion event to the n8n trigger endpoint (if configured) and writes a run summary to `staleness_index.json`.

## Component 2 — CUBEJS AGENT

### What It Generates

For each group, the CubeJS Agent produces three types of files:

**1. `{GoldTableName}.js` — one per gold table in scope**

Maps a single DBT Gold table to a Cube.js Cube definition. Contains:
- `sql_table` reference
- `measures` — derived from the metric catalog (metric_id → SQL aggregate expression)
- `dimensions` — columns not used as measures, typed (string/number/time/boolean)
- `joins` — inferred from foreign key columns and naming patterns
- Cube-level `description` from the metric catalog domain context

**2. `pre_aggregations.js` — one per group**

Defines Cube.js pre-aggregations for dashboard performance. Derived from:
- The chart specs in the layout spec (which metrics appear in posture strip, which in charts)
- Strip cells → `rollup` pre-aggs with `refreshKey: {every: '1 hour'}`
- Chart measures → `originalSql` pre-aggs with grain matching the dashboard `timeframe`

**3. `schema.yml` — one per group**

The Cube.js YAML config entry point that imports all `.js` files for this group. Also defines the `dataSource` (DuckDB or warehouse config reference) and any global filters.

### CubeJS Agent Internal Flow

```
Group input:
  - layout_spec (template, panels, strip_cells, chart_specs)
  - metric_groups (from resolution_payload.resolved_metric_groups)
  - control_anchors (from resolution_payload.control_anchors)
  - gold_tables (list with schema definitions)

Step 1 — CUBE DISCOVERY
  For each gold_table in scope:
    Retrieve similar past Cube schemas from CubeJS Pattern Store
    (query: domain + table_type → returns top-2 matching schemas)

Step 2 — MEASURE EXTRACTION
  For each metric in metric_groups:
    Map metric_id → {sql_expression, type (sum/count/avg/ratio), format}
    Source: Metric Catalog (static map + LLM fill for unknown metrics)

Step 3 — DIMENSION EXTRACTION
  For each gold_table column not claimed as a measure:
    Infer dimension type from column name and SQL type
    Mark time dimensions for grain-based filtering

Step 4 — JOIN INFERENCE
  For each pair of gold_tables in the group:
    Check for FK patterns (e.g., `user_id` appears in both)
    If found → add `joins` block with `relationship` type

Step 5 — PRE-AGG PLANNING
  From chart_specs:
    Strip cells → rollup pre-agg (all measures, no dimensions)
    Time-series charts → rollup pre-agg with time_dimension grain
    Bar/compare charts → rollup pre-agg with category dimension

Step 6 — FILE GENERATION
  LLM generates each .js file from structured context (Prompt 2)
  LLM generates schema.yml (Prompt 3)
  pre_aggregations.js is deterministically assembled from Step 5
```

### CubeJS Pattern Store

**Purpose:** Stores successful CubeJS cube schemas so the agent can retrieve relevant patterns rather than generating from scratch.

**Schema per entry:**
```json
{
  "id": "uuid",
  "domain": "ld_training",
  "table_type": "lms_completion",
  "source_table": "gold.csod_course_completion",
  "cube_name": "CsodCourseCompletion",
  "measures": [...],
  "dimensions": [...],
  "embedding_text": "LMS training completion Cornerstone course assignment user department..."
}
```

**Retrieval:** At Step 1, embed `{domain} {table_name} {metric_group_names}` and retrieve top-2 by cosine similarity. These are injected as few-shot examples into Prompt 2.

**Write-back:** When a generated cube file passes `validation_node`, it is added to the CubeJS Pattern Store for future retrieval.

## Component 3 — N8N AGENT

### What It Generates

A single `n8n_workflow.json` per group — a complete n8n workflow export that can be imported directly into n8n. The workflow has four functional sections:

**1. Trigger Section**
- `Schedule Trigger` — cron expression derived from dashboard `timeframe`
  - daily → `0 6 * * *`
  - weekly → `0 8 * * 1`
  - monthly → `0 9 1 * *`
- Optional: `Webhook Trigger` — for on-demand refresh via API call

**2. Data Fetch Section**
- `HTTP Request` node → calls CubeJS API for each metric group
  - Uses the measure names from the generated CubeJS schemas
  - Applies the dashboard `timeframe` as the date range filter
- `Code` node → normalizes API responses into a flat metric snapshot

**3. Alert Evaluation Section**
- One `IF` node per control anchor with a defined threshold
  - Threshold values come from `semantics.thresholds` in the EPS IntentSpecs
  - `good_direction` (up/down/neutral) determines the comparison operator
- Branches into `Alert` path (threshold breached) or `Pass` path

**4. Output Section**
- Alert path → `Slack` node (send alert to configured channel) + optional `Email` node
- Pass path → `HTTP Request` node → POST metric snapshot to a dashboard refresh endpoint
- Both paths → `Code` node → write run log entry to `staleness_index.json`

### N8N Agent Internal Flow

```
Group input:
  - control_anchors + thresholds (from EPS IntentSpecs in layout_spec)
  - timeframe (from resolution_payload)
  - domain (for schedule and output routing defaults)
  - cubejs_artifact (measure names for data fetch nodes)
  - insights (for context in Slack alert messages)

Step 1 — TRIGGER PLANNING
  Map timeframe → cron expression (deterministic)
  Determine trigger type: schedule | schedule+webhook

Step 2 — DATA FETCH PLANNING
  For each metric_group in resolved_metric_groups:
    Build CubeJS query spec: {measures[], dimensions[], timeDimensions[]}
    Source measure names from cubejs_artifact generated in Component 2

Step 3 — ALERT CONDITION EXTRACTION
  For each control_anchor with threshold defined in EPS IntentSpec semantics:
    Extract: {metric_id, threshold_value, good_direction, control_id}
    Build: IF condition expression for n8n

Step 4 — NODE ASSEMBLY
  Retrieve similar past n8n workflow patterns from n8n Pattern Store
  (query: domain + trigger_type → returns top-1 matching workflow skeleton)
  LLM fills in node parameters using the planned data from Steps 1–3 (Prompt 4)

Step 5 — WORKFLOW VALIDATION
  Check: all IF nodes have both true/false branches
  Check: at least one Trigger node exists
  Check: all HTTP Request nodes have url parameter
  Check: node IDs are unique UUIDs
```

### N8N Pattern Store

**Purpose:** Stores working n8n workflow skeletons by domain and trigger type. The agent retrieves a skeleton and fills in domain-specific parameters rather than generating the full JSON structure from scratch.

**Schema per entry:**
```json
{
  "id": "uuid",
  "domain": "security_operations",
  "trigger_type": "schedule",
  "timeframe": "daily",
  "node_count": 10,
  "alert_conditions_count": 3,
  "workflow_skeleton": {...},   // n8n workflow JSON with {{PLACEHOLDER}} variables
  "embedding_text": "security SOC alert daily schedule CubeJS compliance threshold..."
}
```

**Retrieval:** Embed `{domain} {trigger_type} {timeframe} {alert_condition_count}` and retrieve top-1. The skeleton replaces 80% of the generative work — the LLM only fills in parameter values.

## Component 4 — VERSION AGENT

### Staleness Detection

The version agent checks three sources for change signals before any generation runs:

```
1. Gold Table Changes
   Source: dependency_graph.json + DBT manifest diff
   Signal: column added/removed/renamed on a gold table
   Bump: MAJOR

2. Metric Catalog Changes
   Source: Metric Catalog delta log
   Signal: new metric_id added to a group's metric_groups
   Bump: MINOR

3. Registry Changes
   Source: control_domain_taxonomy.json hash comparison
   Signal: new control added to a framework
   Bump: MINOR (if anchor affects this group)

4. Threshold/Label Changes
   Source: EPS IntentSpec semantics diff
   Signal: threshold value or metric label changed
   Bump: PATCH

5. Manual Trigger
   Source: orchestrator run with --force flag
   Signal: always generate
   Bump: PATCH (unless inputs changed, then follows rules above)
```

### Version Registry Files

**`artifact_versions.json`**
```json
{
  "soc2_audit_hybrid_compliance": {
    "version": "1.1.0",
    "hash": "sha256:abc123",
    "updated_at": "2025-03-05T14:22:00Z",
    "artifacts": {
      "layout_spec": "sha256:def456",
      "cubejs": "sha256:ghi789",
      "n8n": "sha256:jkl012"
    }
  }
}
```

**`dependency_graph.json`**
```json
{
  "gold.csod_course_completion": [
    "lms_learning_target_ld_training",
    "soc2_audit_hybrid_compliance"
  ],
  "gold.snyk_vuln_findings": [
    "soc2_audit_hybrid_compliance",
    "operational_monitoring_security_operations"
  ]
}
```

This graph enables **cascade detection**: when a gold table schema changes, all dependent groups are flagged for MAJOR bump in a single pass.

## Scheduling System

### Scheduler State

The scheduler maintains a priority queue of pending group runs. It is driven by three input channels:

```
Channel A — Change signals (continuous)
  DBT manifest changes → dependency_graph.json lookup → queue affected groups
  Metric catalog updates → flag groups with that metric_group
  Control taxonomy updates → flag all groups with that framework

Channel B — Scheduled sweeps (cron)
  Daily: PATCH-eligible groups (thresholds, labels)
  Weekly: MINOR-eligible groups (new metrics)
  On-demand: MAJOR-eligible groups (schema changes — run immediately)

Channel C — Manual triggers (API)
  POST /orchestrator/run/{group_id} → queue single group
  POST /orchestrator/run-all → queue all groups
  POST /orchestrator/run-domain/{domain} → queue all groups in a domain
```

### Priority Queue Ordering

```python
priority_score = (
    bump_type_weight[bump_type]    # MAJOR=100, MINOR=50, PATCH=10
  + control_anchor_count * 5       # more compliance surface = higher priority
  + days_since_last_run            # aging factor
  + (1 if force_flag else 0) * 200 # manual always jumps queue
)
```

## New Vector Stores — Schemas

### CubeJS Pattern Store

**Store type:** FAISS or ChromaDB
**Embedding model:** Same as existing template store
**Entry count at launch:** Seed with 1 example per domain (8–10 entries)
**Growth:** Write-back on every successful generation pass

```python
class CubeJSPatternEntry:
    id: str                    # uuid
    domain: str                # from dashboard_domain_taxonomy
    table_type: str            # inferred from gold table name (lms_completion, vuln_finding, etc.)
    source_table: str          # gold.{table_name}
    cube_name: str             # PascalCase
    measures: list[dict]       # [{name, sql, type, description}]
    dimensions: list[dict]     # [{name, sql, type}]
    joins: list[dict]          # [{cube, sql, relationship}]
    metric_groups: list[str]   # which metric_groups this cube serves
    embedding_text: str        # concatenated domain + table + metric names for embedding
```

### N8N Pattern Store

**Store type:** FAISS or ChromaDB
**Entry count at launch:** Seed with 1 skeleton per domain + trigger type combination (12–15 entries)
**Growth:** Write-back on every successful generation pass

```python
class N8NPatternEntry:
    id: str
    domain: str
    trigger_type: str          # schedule | webhook | schedule+webhook

    embedding_text: str
```

**Store type:** FAISS or ChromaDB

## Integration with Existing Registries

The Artifact Orchestrator is a **read-only consumer** of all existing registries. It never modifies them.

| Existing Registry | How It's Used |

## End-to-End Sequence

```
STEP 1 — RBSRV pipeline completes VERIFY
  → Approved layout_spec written to working state
  → Orchestrator notified via event (or polled)

STEP 2 — DEPENDENCY CHECK
  → Load group from group_manifest.json
  → Check staleness_index and dependency_graph
  → Determine bump_type
  → If no changes and not forced → skip

STEP 3 — FAN-OUT GENERATION (parallel)
  → layout_node: tag and copy layout_spec
  → cubejs_node: generate cube files (Prompts 2, 3)
  → n8n_node: generate workflow (Prompt 4)

STEP 4 — VALIDATION
  → Validate all three artifacts
  → If any fail → error_node (log, skip group, add to failed_groups)

STEP 5 — VERSION + CHANGELOG
  → Bump semver per rules
  → Generate CHANGELOG entry (Prompt 6)
  → Write manifest.json

STEP 6 — STORAGE
  → Write versioned directory
  → Update symlink and registry
  → Git commit

STEP 7 — NOTIFY
  → Post completion event
  → Update staleness_index (next run due = now + schedule interval)

STEP 8 — NEXT GROUP
  → Pop next group from priority queue
  → Repeat from STEP 3
  → When queue empty → orchestrator run complete
```
