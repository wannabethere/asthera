Remaining Bugs — Require Fixes
🔴 BUG 1 — TC10: json_agg FILTER missing ::JSONB cast
CRITICAL · Runtime Error TC10 step_2 → step_3

The bucket_counts pattern that fixed Bug B in TC1–TC9 was not applied in TC10. TC10 uses json_agg(completed_date) FILTER (...) which returns JSON type. The UDF compare_periods(JSONB, JSONB) cannot accept JSON arguments. PostgreSQL throws: "function compare_periods(json, json) does not exist".

-- ❌ TC10 step_2 GENERATED (fails at step_3 UDF call) SELECT division, json_agg(completed_date) FILTER (WHERE completed_date < ...) AS first_3_months, -- JSON type json_agg(completed_date) FILTER (WHERE completed_date >= ...) AS last_3_months -- JSON type FROM step_1_output GROUP BY division
✅ Fix — add ::JSONB cast to both FILTER aggregates
SELECT division, json_agg(completed_date) FILTER (WHERE completed_date < DATE_TRUNC('month', CURRENT_DATE) - INTERVAL '3 months')::JSONB AS first_3_months, json_agg(completed_date) FILTER (WHERE completed_date >= DATE_TRUNC('month', CURRENT_DATE) - INTERVAL '3 months')::JSONB AS last_3_months FROM step_1_output GROUP BY division

Root cause: The nl_question_spec must_include for TC10 step_2 only specifies the FILTER pattern — it does not include a ::JSONB cast requirement. The general rule "always cast json_agg to JSONB" must be encoded in the NL generator system prompt, not just relied upon per-step nl_question_spec. Alternatively, use jsonb_agg() everywhere instead of json_agg()::JSONB.

🔴 BUG 2 — TC9: Hallucinated training_title = 'cornerstone' filter → 0 rows
CRITICAL · Logic Error · Zero Output TC9 step_1 only

The user query says "...for cornerstone over the last 6 months". The word "cornerstone" is the project_id — it refers to the data source (Cornerstone OnDemand LMS), not a filter value. The planner incorrectly interpreted "cornerstone" as a training_title column value and added AND training_title = 'cornerstone' to the WHERE clause. The table has no rows with training_title = 'cornerstone' → the entire pipeline returns 0 rows.

-- ❌ TC9 step_1 GENERATED SELECT division, completed_date FROM csod_training_records WHERE is_completed = 'true' AND completed_date >= DATE_TRUNC('month', CURRENT_DATE) - INTERVAL '6 months' AND completed_date < DATE_TRUNC('month', CURRENT_DATE) AND training_title = 'cornerstone' -- "cornerstone" is the project_id, not a value -- ✅ CORRECT step_1 (remove spurious filter) SELECT division, completed_date FROM csod_training_records WHERE is_completed = 'true' AND completed_date >= DATE_TRUNC('month', CURRENT_DATE) - INTERVAL '6 months' AND completed_date < DATE_TRUNC('month', CURRENT_DATE)

Root cause: The planner is confusing the project context identifier ("cornerstone") with a filterable column value. The project_id should be stripped from the NL question before it reaches the pipeline planner, or the planner must be instructed: "The phrase 'for [project_id]' indicates the data source context, not a WHERE clause filter. Do not add a training_title, curriculum_title, or any column filter based on the project_id word."

Warnings — Non-Breaking But Should Be Addressed
🟡 WARNING 1 — TC10: NULL counts when a division has no completions in one period
DATA QUALITY TC10 compare_periods output

When json_agg(completed_date) FILTER (WHERE ...) matches no rows for a given division, it returns NULL (not an empty array []). The UDF compare_periods(NULL, JSONB) then returns NULL counts and a NULL difference — which is misleading (should be 0).

-- Actual output when Marketing has no data in first_3_months period: division | first_3_months_count | last_3_months_count | difference -----------+----------------------+---------------------+----------- Marketing | NULL | 1 | NULL -- Fix option 1: COALESCE at the aggregation level COALESCE(json_agg(completed_date) FILTER (...)::JSONB, '[]'::JSONB) AS first_3_months -- Fix option 2: Handle NULL in the UDF implementation
🟡 WARNING 2 — TC4: ORDER BY time_period references column not in SELECT list
STYLE / PORTABILITY TC4 final SELECT only

TC4 final SELECT does not include time_period in its column list (only scalar trend stats + anomaly fields), but the ORDER BY clause uses ORDER BY division, time_period. PostgreSQL resolves time_period from the FROM clause context (j0.time_period from step_4_output). This executes correctly in PostgreSQL 16 but is non-standard SQL and would fail in stricter engines.

-- ⚠ Generated (works in PG16, non-standard) ORDER BY division, time_period -- time_period not in SELECT but resolves from j0 context -- ✅ Explicit (fully portable) ORDER BY p.division, j0.time_period
