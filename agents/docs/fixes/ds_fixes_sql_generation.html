🔴 BUG A — Nested Aggregate in json_agg: COUNT(*) inside json_build_object
CRITICAL · Runtime Error Affects: TC2, TC3, TC4 step_2, TC5 step_2, TC6, TC7, TC8, TC9

Every test case where step_2 counts events per time bucket generates this pattern. PostgreSQL does not allow an aggregate function (COUNT(*)) inside another aggregate function (json_agg). This causes an immediate ERROR: aggregate function calls cannot be nested at runtime.

-- ❌ GENERATED (fails in TC2, TC5, TC6, TC7, TC8, TC9) SELECT division, json_agg(json_build_object('time', completed_date, 'value', COUNT(*))) AS metric_series FROM step_1_output GROUP BY division;
✅ Correct Pattern — Pre-aggregate then json_agg
-- Step 2a: count per (group, time_bucket) — no nesting WITH daily_counts AS ( SELECT division, completed_date, COUNT(*) AS cnt FROM step_1_output GROUP BY division, completed_date ), -- Step 2b: collect into ordered JSONB array step_2_output AS ( SELECT division, json_agg(json_build_object('time', completed_date, 'value', cnt) ORDER BY completed_date)::JSONB AS metric_series FROM daily_counts GROUP BY division )

The NL question generator must emit two must_include items: one for the COUNT sub-aggregation step and one for the json_agg collection step. The planner should split step_2 into a sub-CTE or specify the two-phase pattern explicitly.

🔴 BUG B — json_agg returns JSON, UDFs expect JSONB — missing cast
CRITICAL · Runtime Error Affects: All test cases with UDF calls (TC1–TC9)

json_agg() returns type JSON, but all UDFs are defined with JSONB parameters. PostgreSQL does not auto-cast JSON → JSONB in function calls, resulting in "function calculate_sma(json, integer) does not exist".

-- ❌ GENERATED (type mismatch — json vs jsonb) json_agg(json_build_object('time', completed_date, 'value', 1)) AS metric_series -- Passes json type → calculate_sma(jsonb, int) → NO MATCH -- ✅ FIX: cast to JSONB at the point of aggregation json_agg(json_build_object('time', completed_date, 'value', cnt) ORDER BY completed_date)::JSONB AS metric_series

Add ::JSONB to every json_agg(...) in step_2. The NL question generator must include this in must_include. Alternatively use jsonb_agg() directly.

Recommended Fixes — Prompt & Code Changes
Fix 1 — DS_NL_QUESTION_GENERATOR: Enforce two-phase step_2 pattern

When step_purpose contains "aggregate" + "count" + "jsonb", must_include must encode the pre-aggregation CTE explicitly:

-- must_include should say: "Use a sub-CTE or subquery to GROUP BY division, time_bucket, COUNT(*) AS cnt FIRST", "Then json_agg(json_build_object(..., cnt) ORDER BY time_bucket)::JSONB", "Do NOT put COUNT(*) inside json_build_object directly", "The ORDER BY must be INSIDE json_agg(), not after GROUP BY"
Fix 2 — DS_PIPELINE_PLANNER: Validate join key existence before emitting final_select.on

Add a rule: before specifying join_steps[N].on, verify that EVERY join key column exists in BOTH primary_step.output_columns and join_steps[N].step.output_columns.

-- New planner rule: "MUST NOT include a column in final_select.join_steps[N].on unless that column appears in BOTH primary_step.output_columns AND join_steps[N].step.output_columns. If the primary step function returns only scalar stats (no time_period), do NOT join on time_period. Use LEFT JOIN on division only, or omit the join."
Fix 3 — ds_sql_generator: Don't inject group_col for no-dimension steps

The _infer_group_columns function returns the first non-JSONB input column. For distribution steps, step_2_output has only one column (the JSONB array) — so infer returns empty. The LATERAL instruction must not prepend s.group_col when group_col is empty.

# In _infer_group_columns: def _infer_group_columns(step_plan): input_cols = step_plan.get("input_columns", []) non_jsonb = [c for c in input_cols if c != "metric_series" and c != "status_distribution"] return non_jsonb[0] if non_jsonb else "" # empty string → skip group_col in SELECT
